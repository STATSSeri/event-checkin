# インシデント対応プロセス

S/PASS（イベント受付システム）における事故・障害・セキュリティインシデント発生時の対応手順を定める。

最終更新：2026-05-19

---

## 1. インシデントの定義と分類

### 1-1. インシデントとは

本サービスにおいて、以下のいずれかに該当する事象を「インシデント」と定義する。

- サービスの全部または一部が利用不能になる事象（障害）
- ユーザーデータの漏えい・滅失・毀損のおそれがある事象（セキュリティインシデント）
- 不正アクセス・不正利用のおそれがある事象
- 個人情報保護法等の法令違反のおそれがある事象

### 1-2. 重大度分類

| レベル | 定義 | 例 |
|---|---|---|
| **P1: 重大** | サービス全停止／個人情報漏えい確認／不正アクセスの被害発生 | 全ユーザーがログイン不可、参加者情報の外部流出 |
| **P2: 高** | 一部機能の停止／セキュリティ脆弱性の発見／不正アクセスの試行検知 | メール送信機能の停止、Critical CVE の検出 |
| **P3: 中** | サービス品質の低下／監視アラート発火 | レスポンス遅延、エラー率上昇 |
| **P4: 低** | 一時的な不具合・問い合わせ | 個別ユーザーの操作不能 |

## 2. 検知

### 2-1. 自動検知

| 監視対象 | 監視ツール | 通知先 |
|---|---|---|
| アプリケーション稼働 | Vercel 標準監視 | 運用担当者 |
| データベース稼働 | Supabase 標準監視 | 運用担当者 |
| エラー率・応答速度 | Vercel Analytics | 運用担当者 |
| 依存ライブラリ脆弱性 | GitHub Dependabot | 運用担当者（週次） |
| 異常な認証試行 | Supabase Auth ログ + Upstash レート制限 | 運用担当者 |

### 2-2. 人手による検知

- ユーザー／参加者からの問い合わせ
- 内部レビュー・コードレビュー
- 第三者脆弱性診断（年 1 回予定）

## 3. 対応フロー

### 3-1. 初動対応（検知から 1 時間以内）

1. **インシデント認知**：通知またはユーザー報告を受領
2. **重大度判定**：P1〜P4 を仮判定
3. **対応責任者の指名**：代表取締役またはその指名する者
4. **初動記録の開始**：時刻、検知経緯、初動判断をログ化

### 3-2. 分析・封じ込め（P1: 2 時間以内 / P2: 4 時間以内）

1. **影響範囲の特定**：
   - 影響を受けるユーザー数
   - 漏えい・毀損したデータの種別と件数
   - 攻撃の場合：手口、痕跡、継続性
2. **封じ込め措置**：
   - 該当機能の一時停止
   - 影響アカウントのロック
   - WAF ルールの更新
   - サブプロセッサーへの連絡
3. **証拠の保全**：ログ、データベースのスナップショット等

### 3-3. ユーザー通知

| 重大度 | 通知タイミング | 通知方法 |
|---|---|---|
| P1 | 検知から 24 時間以内 | 電子メール（影響を受けたユーザー全員） |
| P2 | 復旧見通しが立った段階で | 電子メール |
| P3 | 復旧後、必要に応じて | サービス内通知 |
| P4 | 個別対応 | 個別連絡 |

通知内容（P1・P2 の場合）：
- 発生日時・検知日時
- 事象の概要
- 影響範囲（推定でも可）
- 現時点の対応状況
- 復旧見込み
- ユーザー側で取るべき措置（あれば）
- 第二報の予定

### 3-4. 復旧（P1: 24 時間以内目標 / P2: 72 時間以内目標）

1. 根本原因の特定
2. 修正のリリース
3. サービス再開の確認
4. 監視強化期間の設定

### 3-5. 事後対応（復旧後 2 週間以内）

1. **事後報告書（Postmortem）の作成**：
   - 事象の時系列
   - 根本原因
   - 検知・対応の経緯
   - 影響範囲（最終確定値）
   - 改善策（短期・長期）
2. **影響ユーザーへの最終報告**：電子メールにて事後報告書の要約を送付
3. **規制当局への報告**：個人情報保護法上の報告義務に該当する場合は、個人情報保護委員会へ速やかに報告
4. **再発防止策の実装**

## 4. エスカレーションフロー

```
[検知者：監視ツール／運用担当者／ユーザー報告]
         ↓
[第一報受信：代表取締役]
         ↓
   ┌──────┴──────┐
   ↓              ↓
[技術対応]     [外部連絡]
- 封じ込め    - ユーザー通知
- 修正        - サブプロセッサー連絡
- 復旧        - 個人情報保護委員会への報告
              - 必要に応じて顧問弁護士・専門家
```

## 5. サブプロセッサー側のインシデント

Supabase、Vercel、Resend、Stripe、Upstash 等のサブプロセッサー側で発生したインシデントについては：

1. 各事業者のステータスページを常時監視
2. 重大事象の場合、当社からユーザーへ転送通知
3. 個人情報に関わる場合は、サブプロセッサーから受領した情報を基に当社が責任を持って通知

各事業者のステータスページ：
- Supabase: https://status.supabase.com/
- Vercel: https://www.vercel-status.com/
- Resend: https://status.resend.com/
- Stripe: https://status.stripe.com/
- Upstash: https://status.upstash.com/
