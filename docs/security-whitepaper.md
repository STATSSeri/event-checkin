# S/PASS セキュリティホワイトペーパー

最終更新：2026-05-19
バージョン：1.0

---

## 1. はじめに

本ドキュメントは、スタッツ株式会社が提供するイベント受付システム「S/PASS」のセキュリティ対策の概要を、評価検討中の企業様向けに技術的観点でまとめたものです。

ご質問は [support@statsworks.tokyo](mailto:support@statsworks.tokyo) までお問い合わせください。

## 2. サービス概要

### 提供形態
- マルチテナント型 SaaS（クラウドサービス）
- 24 時間 365 日提供（計画停止・障害時を除く）

### 主要機能
- イベント設定・招待状送付
- 参加可否回答（RSVP）の受付
- QRコード発行
- 当日受付・来場記録管理

### 取扱情報の種別
- ユーザー（主催者）のアカウント情報
- イベント情報
- 参加者情報（氏名、メールアドレス、所属組織、出欠ステータス、来場記録）
- 決済情報（Stripe 経由で処理。クレジットカード番号は当社では保持しません）

## 3. インフラ・アーキテクチャ

### 3-1. クラウド構成

サーバーレス（マネージドサービス中心）の構成により、インフラ層の攻撃面を最小化しています。

```
[ユーザー]
   ↓ HTTPS / TLS 1.2-1.3
[Vercel Edge Network] ← WAF / DDoS 緩和 / CDN
   ↓
[Next.js Application] (サーバーレス関数)
   ↓
┌──────────────────────────────────┐
│ Supabase (AWS 東京リージョン)    │
│  - PostgreSQL (RLS + at-rest 暗号化) │
│  - 認証 (TOTP MFA 対応)         │
│  - 自動バックアップ              │
└──────────────────────────────────┘
   ↓ 外部 API
[Resend (メール)] [Stripe (決済)] [Upstash (レート制限)]
```

### 3-2. データ所在地

- **データベース／認証／ストレージ**：AWS 東京リージョン（ap-northeast-1）
- **アプリケーション実行環境**：Vercel グローバルエッジ（東京近傍を優先）
- **バックアップ**：AWS 東京リージョン内のマルチ AZ 構成

## 4. データ保護

### 4-1. 暗号化

| 対象 | 方式 |
|---|---|
| 通信（クライアント↔サーバ） | TLS 1.2 / 1.3、HSTS 強制（max-age=63072000、includeSubDomains、preload） |
| データベース at-rest | AES-256（Supabase 標準） |
| バックアップ | AES-256 |
| パスワード | bcrypt ハッシュ化（Supabase Auth）+ 履歴用に scrypt ハッシュ化 |
| 多要素認証シークレット | Supabase Auth による暗号化保管 |

### 4-2. バックアップ

- Supabase 標準の自動バックアップ（日次）
- Point-in-Time Recovery（PITR）対応
- 保管期間：7 日（Pro プラン）／最大 30 日（Team 以上）
- AWS 東京リージョン内のマルチ AZ で分散保管
- 重要な構成変更前および年 1 回のリストアテスト実施予定

### 4-3. データの削除

契約終了後のデータ取扱いは [データ保管・廃棄ポリシー](./data-retention-policy.md) を参照してください。

## 5. アクセス制御・認証

### 5-1. ユーザー認証

- メールアドレス + パスワードによる認証（Supabase Auth）
- パスワードポリシー：
  - 最低 10 文字以上
  - 大文字・小文字・数字・特殊文字のうち 3 種類以上を含む
- パスワード履歴：直近 10 回分の再利用を禁止
- パスワード最低有効期間：1 日（連続変更の防止）
- 多要素認証：TOTP 方式（Google Authenticator、1Password 等の認証アプリと互換）

### 5-2. アクセス権限制御

- Supabase Row Level Security（RLS）によりデータベースレベルで主催者ごとに分離
- アプリケーション層でも `verifyEventOwnership()` 等の認可チェックを各 API で実施
- 認可ロジックを `src/lib/auth.ts` に一元化

### 5-3. ブルートフォース対策

- Supabase Auth 標準のレート制限（30 回／5 分）
- アプリケーション層のレート制限（認証 API：5 回／60 秒、メール送信 API：30 回／60 秒 等）
- Upstash Redis ベースのレート制限を全主要 API に適用

## 6. ネットワークセキュリティ

### 6-1. 多層防御

| 層 | 対策 |
|---|---|
| エッジ | Vercel WAF、DDoS 緩和、グローバル CDN |
| アプリケーション | セキュリティヘッダー一式、入力検証、CSRF 対策 |
| データベース | Supabase RLS、暗号化通信、IP 制限（必要に応じて） |

### 6-2. セキュリティヘッダー

全レスポンスに以下のヘッダーを設定（[next.config.mjs](../next.config.mjs)）：

- `Content-Security-Policy`：許可するスクリプト／スタイル／接続先を制限
- `Strict-Transport-Security`：HTTPS 強制（2 年間）
- `X-Frame-Options: DENY`：iframe 埋め込み禁止
- `X-Content-Type-Options: nosniff`：MIME スニッフィング防止
- `Referrer-Policy: strict-origin-when-cross-origin`：Referrer 情報最小化
- `Permissions-Policy`：不要なブラウザ機能を無効化

### 6-3. SQL インジェクション対策

- Supabase クライアント経由のパラメタライズドクエリのみ使用
- 生 SQL 文字列の連結は実装上禁止

## 7. 運用セキュリティ

### 7-1. 脆弱性管理

- GitHub Dependabot による依存ライブラリの週次脆弱性スキャン
- 重大脆弱性（Critical）：48 時間以内に対応
- High：1 週間以内
- Medium：月次定期更新
- インフラ側パッチは Vercel / Supabase が自動適用

### 7-2. 不変インフラストラクチャ

- 各デプロイは独立したコンテナで実行（イミュータブル）
- サーバ上のファイル改ざんリスク自体を排除

### 7-3. 環境分離

- 本番環境と開発環境を完全分離
- 環境変数によるシークレット管理（Vercel 環境変数）

## 8. ログ・監査

### 8-1. 取得しているログ

| 種別 | 取得内容 | 保管期間 |
|---|---|---|
| 認証ログ | ログイン成功／失敗、サインアップ、パスワード変更 | 90 日（Supabase 標準） |
| アクセスログ | リクエストパス、応答ステータス、IP アドレス | 90 日（Vercel 標準） |
| アプリケーションログ | エラー、警告 | 90 日 |
| パスワード変更履歴 | ハッシュ化された過去 10 回分 | 履歴更新まで |

### 8-2. ログへのアクセス制御

- ログは Supabase / Vercel の管理コンソール経由でのみアクセス可能
- アクセス権限は当社の限定された運用担当者のみ

## 9. インシデント対応

詳細は [インシデント対応プロセス](./incident-response.md) を参照してください。

- 24 時間 365 日の自動監視（Vercel / Supabase）
- 重大障害発生時は検知から 24 時間以内に対象ユーザーへ電子メール通知
- 通知内容：障害発生時刻、原因、影響範囲、復旧見込み、恒久対応策
